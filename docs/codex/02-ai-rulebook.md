# AI Rulebook

This file controls how AI agents should build Harmonic OS.

## First Rule

Before writing code, read the Codex files in `/docs/codex/`.

Do not invent a new brand direction. Do not replace the Harmonic OS concept with a generic portfolio, agency site, creator page, or Linktree clone.

## Product Identity Rules

- Harmonic OS is a creator operating system.
- The Cup is the origin object.
- The Harmonic Heart is the soul/core symbol.
- Purple neon represents living frequency.
- The Four Worlds are Melodic, 2 Harmonic, Fried Em, and Schmackinn.
- The phrase **One Frequency. Many Worlds.** is the central tagline.

## Design Rules

- Use black/near-black as the main environment.
- Use purple neon as the primary energy color.
- Use glassmorphism for panels, cards, overlays, and OS windows.
- Use chrome/silver accents carefully.
- Keep layouts premium and spacious.
- Mobile must feel intentional, not like a compressed desktop site.
- Avoid rainbow RGB gaming aesthetics.
- Avoid basic bootstrap/card-template layouts.

## Animation Rules

Every animation must have meaning.

Preferred animation language:

- Cup pour.
- Frequency waves.
- Heart build/reveal.
- Portal openings.
- Glass panels glowing on hover.
- Soft page fades.
- World switch transitions.
- Boot sequence loading.

Avoid:

- Random bouncing.
- Cheap glitch effects everywhere.
- Overloaded motion.
- Animations that hurt readability.

## Code Rules

- Build modular components.
- Separate content data from UI whenever possible.
- Use reusable design tokens for colors, spacing, fonts, and motion.
- Keep public content easy to edit later from the Control Hub.
- Prefer accessible HTML and readable semantic structure.
- Do not hardcode social links in many places; centralize them.

## Copilot Task Rules

When assigned a task:

1. Read relevant Codex files.
2. Confirm the intended system/version.
3. Build only the requested scope.
4. Do not overbuild future systems prematurely.
5. Document assumptions.
6. Keep the OS identity consistent.
